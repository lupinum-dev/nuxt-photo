# SSR and Performance

nuxt-photo ships three distinct SSR modes. Pick the one that matches your constraints — zero-CLS is a guarantee you opt into, not a blanket claim.

::responsive-playground{mode="ssr"}
::

## The three SSR modes

| Mode | How you enable it | Guarantee |
|---|---|---|
| **Approximate fallback** | No `defaultContainerWidth`, no `breakpoints`, no responsive inputs | Reasonable first paint; layout may reflow on hydration. |
| **Exact single-width** | Set `defaultContainerWidth` | SSR matches hydration when the client container lands at that width. |
| **Exact breakpoint-aware** | Set `breakpoints` (or use responsive `columns` / `spacing` / `padding` / `targetRowHeight`) | SSR matches hydration across every width that falls inside a defined span. |

**Precedence** (columns / masonry): breakpoint-aware → exact single-width → approximate fallback. Explicit breakpoints always win. Responsive inputs imply breakpoints via `mergeResponsiveBreakpoints`, so you don't have to pass `breakpoints` manually when you already pass a `responsive()` value.

Rows always supports all three modes. Columns and masonry render per-breakpoint snapshots — one DOM copy per span, gated by `@container` queries — so the server-rendered layout matches the hydrated one at every defined width.

## The Problem

Photo layouts depend on the container width — but during SSR, there's no DOM to measure. A naive approach renders a placeholder on the server, then re-renders with the correct layout after mount, causing visible layout shift.

## Zero-CLS Strategy for Rows

The rows layout uses a two-phase approach:

### Phase 1: CSS Flex-Grow Fallback (Pre-Mount)

On the server and before the client mounts, photos are rendered in a flex container. Each photo gets a `flex-grow` value proportional to its aspect ratio, so the browser lays them out in justified rows purely with CSS — no JavaScript needed.

This isn't pixel-perfect (CSS flex-grow doesn't know about the target row height or optimal line breaks), but it produces a reasonable approximation that prevents CLS.

### Phase 2: JS-Computed Layout (Post-Mount)

After mount, the `ResizeObserver` measures the actual container width. The Knuth-Plass algorithm computes the optimal row breaks, and inline styles replace the flex-grow fallback. The elements are the same DOM nodes — only their styles change.

## defaultContainerWidth + breakpoints

For pixel-perfect server rendering, tell the layout what width to expect:

```vue
<PhotoAlbum
  :photos="photos"
  :default-container-width="1200"
  :breakpoints="[375, 600, 900, 1200]"
/>
```

When `defaultContainerWidth` is set (and neither `breakpoints` nor responsive inputs are supplying a breakpoint-aware snapshot path):
1. The JS layout runs on the server with that width
2. The HTML includes exact widths (not flex-grow approximations)
3. On the client, the container width is measured and snapped to a breakpoint
4. If the snapped width matches the server width → zero CLS

::callout{type="info"}
Choose a `defaultContainerWidth` that matches one of your `breakpoints` for the most common viewport. For a full-width album on desktop, `1200` is a good default.
::

## Breakpoint Snapping

The `breakpoints` prop snaps the observed container width to the largest breakpoint ≤ actual width:

```vue
<PhotoAlbum :photos="photos" :breakpoints="[375, 600, 900, 1200]" />
```

| Actual width | Snapped width |
|---|---|
| 1440px | 1200px |
| 1050px | 900px |
| 620px | 600px |

**Benefits:**
1. **SSR match:** Server and client agree on the width when both use the same breakpoints
2. **Fewer re-layouts:** Small width changes (resizing, scrollbar toggling) don't trigger recalculation
3. **Scrollbar oscillation prevention:** When content causes a scrollbar to appear/disappear (~15px width change), snapping absorbs the fluctuation

## Container Queries (Rows Layout)

For the rows layout, nuxt-photo can generate CSS container query rules that handle all breakpoint widths from the first paint:

1. The system computes the optimal layout at each breakpoint width
2. It generates `@container` rules with the exact item widths for each breakpoint
3. These rules are injected as a `<style>` element inside the album

This means the layout is correct **before JavaScript runs** — even before hydration. After mount, JS takes over.

This happens automatically when `breakpoints` are provided.

## Columns / Masonry SSR

Columns and masonry render per-breakpoint snapshots. The server emits **one DOM copy per breakpoint span**, each produced by running the real layout algorithm at that span's sample width. A small stylesheet of `@container` rules toggles which snapshot is visible:

```css
@container np-XXXX (min-width: 0) { .np-album__bp-snapshot { display: none } }
@container np-XXXX (min-width: 600px) and (max-width: 1119px) {
  .np-album__bp-snapshot[data-bp="600-1119"] { display: flex }
}
```

Adjacent breakpoints are merged into a single span only when **group assignment, resolved `spacing`, and resolved `padding` all match**. Any difference produces a separate snapshot — so responsive `spacing`, `padding`, or `columns` automatically increase the number of snapshots without you having to list breakpoints explicitly.

**To opt in:**

```vue
<PhotoAlbum
  :photos="photos"
  layout="masonry"
  :breakpoints="[375, 600, 900, 1200]"
/>
```

Or let the library infer breakpoints from a responsive prop:

```vue
<PhotoAlbum
  :photos="photos"
  :layout="{ type: 'columns', columns: responsive({ 0: 1, 640: 2, 1120: 3 }) }"
/>
```

If you set neither, columns/masonry fall through to the approximate flat grid. In development you'll see a one-time console warning pointing back to this guide.

## Performance Tips

- **Use `breakpoints`** — they reduce re-layout frequency and enable SSR matching
- **Use `loading="lazy"`** — this is the default; images below the fold load on scroll
- **Provide `thumbSrc`** on your photos — smaller thumbnails load faster than full-resolution images
- **Use `@nuxt/image`** — it generates responsive srcsets automatically, so the browser loads the right size
- **Set `sizes`** on PhotoAlbum — helps the browser choose the right image from the srcset before layout completes

```vue
<PhotoAlbum
  :photos="photos"
  :sizes="{ size: '100vw', sizes: [{ viewport: '(min-width: 1024px)', size: '80vw' }] }"
/>
```

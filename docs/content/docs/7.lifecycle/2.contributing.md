Use the contributor guide when you are working in the repo instead of consuming the published package.

Nuxt Photo keeps the full contributor workflow in the root [`CONTRIBUTING.md`](https://github.com/lupinum-dev/nuxt-photo/blob/main/CONTRIBUTING.md). That file covers:

- workspace layout
- local development commands
- required checks before review
- when code changes require doc updates
- where the active docs live in the repo

## Contributor workflow at a glance

Most changes follow the same path:

1. Work in the smallest package or app that proves the behavior.
2. Update public docs in the same change when behavior, API, defaults, or examples moved.
3. Run the narrowest checks while iterating.
4. Run the full relevant gate before review.

## Before you open a pull request

At minimum:

- work against the smallest package or app that proves the change
- update docs in the same change when public behavior moved
- run the relevant verification gate before review

## Verification gate

The normal pre-review checks are:

```bash
pnpm lint
pnpm typecheck
pnpm test:unit
pnpm test:module-package
pnpm size
```

Use `pnpm build:docs` when you changed the docs app, docs content, or shared components that affect docs rendering.

## Docs source of truth

Public docs live in `docs/content/docs/**`. Root docs like `README.md`, `CHANGELOG.md`, and `CONTRIBUTING.md` stay selective and support the GitHub front door, release history, and contributor workflow.

Because the docs app search indexes the `docs` collection, the public docs pages in this lifecycle section should carry the important contributor guidance themselves instead of acting as empty signposts.

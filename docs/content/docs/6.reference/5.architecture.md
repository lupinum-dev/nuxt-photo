This page orients new contributors. It's deliberately short — the code is the
source of truth — but a few things are worth saying once, out loud, that you
can't derive by reading a single file.

## Package layering

Nuxt Photo is published as a stack of packages. Each layer only depends on
the layer below it. When you add something, place it at the lowest layer that
can hold it.

```
┌─────────────────────────────────────────────┐
│  @nuxt-photo/nuxt       (auto-imports,      │  module
│                          SSR glue)          │
├─────────────────────────────────────────────┤
│  @nuxt-photo/recipes    (PhotoAlbum,        │  opinionated
│                          PhotoCarousel, …)  │  compositions
├─────────────────────────────────────────────┤
│  @nuxt-photo/vue        (composables,       │  reactive
│                          primitives, SFCs)  │  bindings
├─────────────────────────────────────────────┤
│  @nuxt-photo/engine     (lightbox lifecycle │  framework-
│                          as a state machine)│  agnostic
├─────────────────────────────────────────────┤
│  @nuxt-photo/core       (layout, geometry,  │  pure logic,
│                          physics, adapters) │  no Vue
└─────────────────────────────────────────────┘
```

- **core** is pure TypeScript. No Vue, no DOM globals at import time. This is
  where Knuth-Plass row layout, masonry, spring physics, gesture
  classification, image adapters, and the transition planner live. Anything
  testable without a browser goes here.
- **engine** is the lightbox lifecycle modelled as a state machine + pure
  event handlers. It doesn't know what Vue is. Another framework could bind
  to it without changes.
- **vue** wires engine state to Vue reactivity, manages DOM refs, owns the
  ghost transition choreography, and exposes composables (`useLightbox`,
  `useLightboxProvider`) and unstyled primitives.
- **recipes** are finished components (`PhotoAlbum`, `PhotoCarousel`,
  `Lightbox`). They compose vue-layer primitives into something you drop in.
- **nuxt** registers auto-imports and handles SSR plumbing.

## Lightbox data flow

```
user click on <PhotoTrigger>
      ↓
useLightbox().open(i)
      ↓
engine.open(i)                   ── updates state machine
      ↓
carousel.goTo(i)                 ── Embla paging
      ↓
ghost.open(i, callbacks)         ── animation choreography
      │
      ├─► planCloseTransition /  ── picks FLIP vs FADE
      │   shouldUseFlip
      │
      ├─► ensureImageLoaded      ── decode() → onload fallback
      │
      └─► resolves when media visible
      ↓
engine.markOpened()
      ↓
Vue reactivity re-renders        ── engineState → <Lightbox>
```

The inverse happens on close, via `planCloseTransition` (see
[packages/core/src/transition/planner.ts](https://github.com/lupinum-dev/nuxt-photo/blob/main/packages/core/src/transition/planner.ts)).

## Design notes

A few decisions that would surprise you reading only one file:

### Why Knuth-Plass for rows layout

Greedy row-packing is fast and often fine, but can trap the final row into
being stubby or unusually tall because earlier rows took too many (or too
few) photos. We use a Knuth-Plass style DP (global minimum of squared
deviation from target row height), which fixes that without resorting to a
heap or graph search — a single `Float64Array` and an `Int32Array` of
pointers is enough. See
[knuthPlass.ts](https://github.com/lupinum-dev/nuxt-photo/blob/main/packages/core/src/layout/rows/knuthPlass.ts).

### Why container queries, not resize observers

The row layout at a given container width is fixed (the divisor in each
item's `width: calc(…)` formula is independent of container width within a
span of breakpoints). We precompute layouts at a set of breakpoints and emit
`@container` rules. Adjacent breakpoints with identical row signatures are
collapsed into a single rule. Net effect: responsive, fluid albums with zero
JS resize listeners and no layout thrash. See
[containerQueries.ts](https://github.com/lupinum-dev/nuxt-photo/blob/main/packages/core/src/layout/rows/containerQueries.ts).

### Why FLIP with a FADE fallback

FLIP (morphing the thumbnail rect into the viewer rect) gives the strongest
sense of continuity — you're pretty sure the photo you clicked is the one
that grew. It's the default. But FLIP from a thumbnail that's off-screen or
clipped to a few pixels looks like a random zoom from nowhere, so the
transition planner falls back to an overlay FADE when the source rect fails
a visibility check. `prefers-reduced-motion` downgrades to FADE as well. See
[planner.ts](https://github.com/lupinum-dev/nuxt-photo/blob/main/packages/core/src/transition/planner.ts).

### Why a dedicated "ghost" element

The FLIP uses a separate absolutely-positioned `<img>` (the ghost) rather
than animating the viewer's own image. Two reasons:

1. The viewer image is only inserted once the full-size source has decoded,
   so it can't carry the transition from the thumbnail.
2. The ghost uses `thumbSrc` (already in cache), so the morph starts
   instantly. The full image decodes in parallel and is revealed by setting
   `mediaOpacity` to 1. There's an invariant the code relies on: that
   `ensureImageLoaded(photo.src)` resolves _before_ `mediaOpacity = 1` on
   every open path. See
   [openTransition.ts](https://github.com/lupinum-dev/nuxt-photo/blob/main/packages/vue/src/composables/ghost/openTransition.ts).

### Why the engine is framework-agnostic

The engine package exists so we can unit-test the whole lightbox lifecycle
without mounting Vue. State transitions are pure; side effects (animations,
preloads) live in the Vue layer behind callbacks. This is also what makes a
future React or Svelte port a surface-level rewrite.

## Where to look first

| You want to …                        | Start in                                                                               |
| ------------------------------------ | -------------------------------------------------------------------------------------- |
| Understand overall orchestration     | `packages/vue/src/composables/useLightboxContext.ts`                                   |
| Change how rows are packed           | `packages/core/src/layout/rows/knuthPlass.ts`                                          |
| Tune the open/close animation        | `packages/vue/src/composables/ghost/openTransition.ts` + `closeTransition.ts`          |
| Change FLIP-vs-FADE rules            | `packages/core/src/transition/planner.ts`                                              |
| Add a new image adapter              | `packages/core/src/image/adapter.ts`                                                   |
| Touch gestures (swipe / pinch / pan) | `packages/core/src/viewer/gestures.ts` + `packages/vue/src/composables/useGestures.ts` |
| Change spring feel                   | `packages/core/src/physics/spring.ts`                                                  |

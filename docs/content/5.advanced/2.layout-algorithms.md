## Rows: Knuth-Plass Dynamic Programming

The rows layout solves a classic line-breaking problem: given N photos with varying aspect ratios, find the optimal places to break them into rows such that each row's height is as close to the target as possible.

### The Problem

Given N photos with known aspect ratios, photos placed side by side in a row share the same height. The wider the photos (in total aspect ratio), the shorter the row. You want every row to be close to a target height `h`.

Formally: partition photos `[0..N-1]` into consecutive groups (rows) such that the total **cost** is minimized:

```
cost(row) = (actualHeight - targetHeight)² × rowLength
```

The multiplication by `rowLength` penalizes large deviations more when more photos are affected.

### The Actual Row Height

```
commonHeight = usableWidth / Σ(aspectRatio_i)
```

where `usableWidth = containerWidth - (n-1) × spacing - 2 × n × padding`.

### Why Knuth-Plass over Dijkstra + Heap?

The classic approach models this as a shortest-path problem — but:

1. **The graph is a DAG.** No need for Dijkstra's relaxation; a simple forward pass works.
2. **No heap overhead.** Dijkstra with a binary heap is O(E log V). The DP approach is O(N·K) with no heap, no priority queue.
3. **Cache-friendly memory.** Two typed arrays (`Float64Array` for costs, `Int32Array` for pointers) — contiguous memory, great cache locality.
4. **K is bounded.** The max photos per row is `⌊containerWidth / (targetRowHeight × minAspectRatio)⌋ + 2` — typically ~10.

The DP gives the **globally optimal** solution with less constant-factor overhead.

### The Algorithm

```ts
const minCost = new Float64Array(N + 1).fill(Infinity)
const pointers = new Int32Array(N + 1).fill(0)
minCost[0] = 0

for (let i = 1; i <= N; i++) {
  const start = Math.max(0, i - limitNodeSearch)
  for (let j = i - 1; j >= start; j--) {
    const rowCost = cost(photos, j, i, containerWidth, targetRowHeight, spacing, padding)
    if (rowCost === undefined) continue

    const totalCost = minCost[j] + rowCost
    if (totalCost < minCost[i]) {
      minCost[i] = totalCost
      pointers[i] = j
    }
  }
}

// Reconstruct path
const path = []
let curr = N
while (curr > 0) {
  path.push(curr)
  curr = pointers[curr]
}
path.push(0)
path.reverse()
// path = [0, 3, 7, 12, N] → rows are photos[0..2], [3..6], [7..11], [12..N-1]
```

### Complexity

| | Time | Space |
|---|---|---|
| Knuth-Plass DP | O(N·K) | O(N) — two typed arrays |
| Dijkstra + heap | O(N·K·log N) | O(N) — heap-allocated nodes |
| Greedy | O(N) | O(1) |

## Columns: Shortest-Path Distribution

Given N photos and C columns, assign consecutive photos to columns such that column heights are as balanced as possible. This is modeled as a shortest path of exactly C edges.

### The Model

- Node 0 = start, Node N = end
- Edge from `j` to `i` = "photos[j..i-1] form one column"
- Edge weight = `(columnHeight - targetHeight)²`
- Find the minimum-weight path of exactly C edges from 0 to N

### The Algorithm

A layer-by-layer BFS through a DAG, where each layer represents one more column:

```ts
// matrix[node][length] = { previousNode, accumulatedWeight }
// Build up C layers, tracking the best predecessor at each (node, length) cell

for (let length = 0; length < C; length++) {
  for (const node of currentQueue) {
    for (const { neighbor, weight } of getColumnNeighbors(node)) {
      const newWeight = accWeight + weight
      if (!existing || existing.weight > newWeight) {
        matrix[neighbor][length + 1] = { node, weight: newWeight }
      }
    }
  }
}

// Reconstruct backwards from (N, C) to (0, 0)
```

This gives the globally optimal column assignment in O(C·N·K).

## Masonry: Greedy + Local Search

### Phase 1: Greedy Placement

Classic masonry — place each photo in the shortest column:

```ts
for (let i = 0; i < photos.length; i++) {
  // Find the shortest column
  let shortest = 0
  for (let c = 1; c < columns; c++) {
    if (colHeights[c] < colHeights[shortest]) shortest = c
  }
  colItems[shortest].push(i)
  colHeights[shortest] += photoHeights[i] + spacing
}
```

O(N·C) — good but not optimal. The bottom edge can be jagged.

### Phase 2: Local Search Optimization

Up to 50 iterations trying to reduce the height difference between tallest and shortest columns. Each iteration tries:
1. **Transfer:** Move a photo from the tallest column to the shortest
2. **Swap:** Exchange a photo between the tallest and shortest columns

```
Greedy only:     ████████████████████    (tall)
                 ████████████            (short)
                 ███████████████

After local search: ██████████████████
                    █████████████████
                    █████████████████    (balanced)
```

Original photo order within each column is preserved by re-sorting indices after the search.

### Why Not Optimal?

Optimal masonry (minimizing max column height) is NP-hard — it reduces to the multiprocessor scheduling problem. Greedy + local search is pragmatic: effectively O(N·C), typically converges in 2–5 iterations.

## Bento: Auto-Span Heuristics

### Auto Sizing

Three steps:

**Step 1:** Meta overrides — if `photo.meta.featured`, `meta.span = '2x2'`, `'wide'`, or `'tall'`, use those spans directly.

**Step 2:** Score remaining photos by aspect ratio deviation from square:

```ts
const score = Math.abs(Math.log(width / height))
// 2:1 landscape → ln(2) ≈ 0.69
// 1:2 portrait  → ln(0.5) ≈ 0.69
// 1:1 square    → 0
```

**Step 3:** Assign spans to top ~30% of photos (no two adjacent):

| Aspect ratio | Span |
|---|---|
| ≥ 1.4 (wide) | 2 columns × 1 row |
| ≤ 0.85 (tall) | 1 column × 2 rows |
| Otherwise | 2 columns × 2 rows |

### Pattern Sizing

A repeating cycle — every `patternInterval`-th photo gets the next element from `['wide', '2x2', 'tall', '2x2']`. Produces a predictable visual rhythm.

### Manual Sizing

Reads `colSpan` and `rowSpan` directly from `photo.meta`. Full control, no heuristics.

## Summary

| Layout | Algorithm | Complexity | Optimality |
|---|---|---|---|
| **Rows** | Knuth-Plass DP | O(N·K) | Globally optimal |
| **Columns** | Shortest path (length C) | O(C·N·K) | Globally optimal |
| **Masonry** | Greedy + local search | O(N·C) | Approximate |
| **Bento** | Heuristic span assignment | O(N log N) | Heuristic |
